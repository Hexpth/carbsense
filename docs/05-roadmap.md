# Roadmap CarbSense (24 месяца × 15 ч/нед)

## Месяц 0–1 (60 ч): Литература

Цель: понимать, где state-of-the-art и почему он там, где есть.

**Прочитать:**
- Dalla Man C. et al. (2007) «Meal Simulation Model of Glucose-Insulin System» — оригинальная UVA/Padova модель.
- Mahmoudi Z. et al. (2019) «Carbohydrate counting via inverse modeling» — близкий baseline.
- Daniels J. et al. (2022) «A multitask learning approach to personalised blood glucose prediction».
- AAPS Autotune / Autosens документация — как сейчас оценивают параметры пациента.
- OpenAPS Data Commons README + примеры использования.

**Артефакт:** документ `docs/06-literature-review.md` с ~20 проаннотированными статьями.

## Месяц 2–3 (120 ч): Физика и симулятор

Цель: уметь генерировать синтетические CGM-кривые из «искусственных пациентов».

- Установить SimGlucose (Python implementation UVA/Padova).
- Понять каждое из 13 уравнений модели — выписать с физическим смыслом.
- Реализовать **свой** упрощённый ODE-solver на NumPy/JAX (4 уравнения — глюкоза в плазме, в interstitial fluid, в кишечнике, инсулин). Это нужно для последующего PINN.
- Сравнить свой solver с SimGlucose на 100 синтетических кейсах.

**Артефакт:** Python модуль `src/carbsense/physiology/`, проходящий тесты на численную идентичность с эталоном на 1%.

## Месяц 4–6 (180 ч): Данные

Цель: чистый, готовый к обучению датасет.

- Скачать OpenAPS Data Commons (~200 пациентов).
- Парсер Nightscout JSON → pandas DataFrame.
- Выделить «meal events»: моменты, где пользователь ввёл значимое количество углеводов (>15 г).
- Чистка: убрать события, перекрывающиеся с другими (boluses, exercise).
- Train/val/test split **по пациентам**, не по времени (иначе утечка).
- Документация датасета: распределения, статистика, baselines.

**Артефакт:** `src/carbsense/benchmark/`, jupyter notebook с EDA, ~10–20 тысяч «чистых» meal events.

## Месяц 7–9 (180 ч): Baselines

Цель: иметь набор baselines, относительно которых будем мерить прогресс.

- **B1**: Линейная регрессия по фичам Δ_30min, Δ_60min, slope, IOB.
- **B2**: Random Forest на тех же фичах.
- **B3**: Extended Kalman Filter на минимальной модели Бергмана (классический академический подход).
- **B4**: LSTM на сырой CGM-последовательности.
- **B5**: Прямая инверсия UVA/Padova через scipy.optimize (gradient-based + регуляризация).

Сравнительная таблица MAE / R² / Recall@40g / FPR.

**Артефакт:** `notebooks/baselines.ipynb` + первый pre-print **препринт на arXiv**: «Carbohydrate Estimation from CGM: A Benchmark of Classical and ML Approaches».

Уже здесь — **первый реальный вклад в сообщество**. Открытый бенчмарк, на который смогут ссылаться другие.

## Месяц 10–15 (360 ч): Physics-Informed Neural Network

Главный вклад проекта.

Идея: вместо «нейросеть напрямую предсказывает C», обучаем сеть **параметризовать априорное распределение** для байесовской инверсии физической модели.

- Реализация PINN на JAX/PyTorch.
- Loss = физическая невязка + ML-регуляризация + supervised термин.
- Подбор архитектуры (Transformer encoder для CGM-окна).
- Cross-validation по пациентам.

**Артефакт:** улучшение MAE до целевых < 12 г. Если не получается — отступаем, документируем причины, ещё ценная статья «What doesn't work and why».

## Месяц 16–18 (180 ч): Персонализация

Без персонализации алгоритм для конкретного пациента — посредственный.

- Few-shot adaptation: 1 неделя данных пациента → fine-tune сети.
- Online learning: каждое новое meal event с известным C обновляет персональную модель.

**Артефакт:** API `model.adapt(patient_history)` + бенчмарк «cold start vs after N events».

## Месяц 19–22 (240 ч): Публикация

- Расширенная версия arXiv-препринта с PINN-результатами.
- Подача в **JMIR Diabetes** или **Diabetes Technology & Therapeutics** или **Journal of Diabetes Science and Technology**.
- Цикл peer-review (3–6 месяцев — может пересечься со следующим этапом).

## Месяц 23–24 (120 ч): Интеграция в AAPS

- Обернуть Python-модель в ONNX → запустить на телефоне.
- Kotlin-обёртка для AAPS.
- PR в `androidaps/main` (предварительно — обсуждение в AAPS Discord/GitHub Issues).
- Документация на русском и английском.

**Артефакт:** код в production у тысяч пользователей.

## Что может пойти не так

| Риск | Mитигация |
|---|---|
| OpenAPS Data Commons недоступен или слишком зашумлён | Запасные датасеты: OhioT1DM, D1NAMO. Запросить доступ к датасетам JDRF. |
| PINN-подход не даёт улучшения | Документировать как negative result — это тоже наука. Откатиться на pure ML с честной публикацией. |
| AAPS PR отвергают | Опубликовать как standalone Python-библиотеку. Любой другой проект (Loop, Strimma) подберёт. |
| Регуляторика | Не делаем медизделие. Делаем «research code». Юридически безопасно. |
| Появляется коммерческий конкурент | Хорошо. Open-source альтернатива всё равно нужна. |
| Выгорание | Каждый артефакт каждые 3 месяца — даёт feedback и поддерживает мотивацию. |

## Что НЕ входит в roadmap

- ❌ Своё мобильное приложение.
- ❌ Свой UI.
- ❌ Управление помпой.
- ❌ Сертификация.
- ❌ Бизнес-модель / стартап.

Это сознательно. Вся ценность — **в алгоритме и публикации**. Распространение — через интеграцию в существующие платформы (AAPS, Loop, xDrip+).
